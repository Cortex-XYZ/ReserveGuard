# ReserveGuard V1

ReserveGuard V1 is a small Solidity library for explicit reserve-aware execution on Monad.

The V1 promise is:

```text
Detect reserve state -> fail earlier at explicit checkpoints
```

## Included

- `IReserveBalance` interface for Monad's MIP-4 precompile at `0x1001`
- `ReserveGuard.dipped()` for reserve-state detection
- `ReserveGuard.assertHealthy()` for fail-fast assertions
- `ReserveGuard.checkpoint()` for deliberate execution boundaries
- `ReserveAware` for reusable internal helpers and `reserveHealthy`
- `ReserveProtected` for functions that must start and finish healthy
- `ReserveViolation()` as the standardized V1 custom error
- Foundry tests with mocked MIP-4 precompile responses
- Developer examples and live Monad testnet experiment contracts
- Optional browser lab for inspecting testnet experiment flows

## Not Included

V1 intentionally does not include:

- automatic recovery policies
- named checkpoints
- checkpoint events
- caller-defined assertion errors
- TypeScript SDK
- monitoring or analytics infrastructure
- generalized account abstraction framework

Those features remain future work until more Monad applications produce practical MIP-4 usage patterns.

## Experimental Work

The repository may contain post-V1 contracts under `contracts/experimental/`. They are deliberately excluded from the V1 compatibility promise:

- `ReserveTrace` explores labeled observations and checkpoints.
- `ReserveRecoverable` explores application-defined recovery and post-recovery verification.

See `docs/experimental/` for their APIs, evidence, and graduation criteria.

## Verification

Local tests validate the Solidity integration shape:

```bash
forge build
forge test
```

If `forge` is not on the shell path, use the installed binary directly:

```bash
~/.foundry/bin/forge test
```

Live Monad testnet transactions remain the authority for reserve-state semantics. The current testnet evidence shows an EIP-7702 delegated EOA moving:

```text
beforeDip = false
duringDip = true
afterDip = false
```

A fresh recoverable EIP-7702 flow also observed `false -> true -> false` around a successful recovery. A later repeated authorization observed `false` during replay even though the on-chain transaction was rejected, so persisted and repeated delegation shapes remain under investigation.

See `docs/live-testnet.md` for the repeatable workflows and `docs/observations/` for dated evidence reports.
